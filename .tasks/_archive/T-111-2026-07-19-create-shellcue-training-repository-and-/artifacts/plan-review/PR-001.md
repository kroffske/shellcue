## Findings

### PR-001 — Model-only base must precede extraction

- Lens / severity: dependencies / medium
- Claim under test: repository extraction can begin from current ShellCue main.
- Evidence: the referenced model-only spec and deterministic-candidate removal
  are introduced by open ShellCue PR #5, not current main.
- Challenge: without an explicit prerequisite, implementation could build new
  training ownership against a runtime base that still permits the invalid
  policy.
- Recommendation: block implementation on PR #5 while allowing this planning
  PR to land independently.
- Planner response: accepted. `task.md` names PR #5 as a base prerequisite and
  delivery sequence starts with its merge.
- Disposition: mitigated

### PR-002 — Cross-repository task can hide partial completion

- Lens / severity: scope and closure / medium
- Claim under test: one task can establish both ShellCue contract authority and
  the new repository without producing an ambiguous half-migration.
- Evidence: the outcome spans two repositories, while each requires a separate
  clean base, PR, and verification surface.
- Challenge: a green ShellCue contract PR alone could be mistaken for completed
  migration even if `shellcue-training` remains dependent on Smart Bash.
- Recommendation: require both independently reviewable delivery slices and a
  Smart-Bash deletion test before closure.
- Planner response: accepted. Work items W2-W7, Verification, and Closure bind
  both slices and reject file-copy/import success as sufficient evidence.
- Disposition: mitigated

### PR-003 — Donor selection can expand after imports are traced

- Lens / severity: blast radius and simplicity / medium
- Claim under test: the named donor inventory is sufficient for a focused
  extraction.
- Evidence: several Smart Bash scripts and evaluators import runtime/model
  modules; exact transitive ownership has not yet been frozen per file.
- Challenge: migration may grow into a disguised fork or silently carry runtime
  behavior into the training package.
- Recommendation: make the donor manifest the first executable gate, classify
  every transitive dependency, and replan on expansion beyond the reviewed
  manifest.
- Planner response: accepted. W1, the per-file migration record, rejected
  classes, and pressure-pass growth triggers impose this gate.
- Disposition: mitigated

### PR-004 — Public repository creation is a separate risk boundary

- Lens / severity: privacy, licensing, and delivery / medium
- Claim under test: creating the sibling repository implies it can immediately
  receive a GitHub remote.
- Evidence: donor code, model/data licenses, private histories, generated runs,
  and historical Git content have different publication constraints.
- Challenge: remote creation before audit could publish private or
  incompatible content even if the package works locally.
- Recommendation: initialize local Git only; require a clean-history,
  large-file, license, provenance, and private-data audit before remote
  creation or public visibility.
- Planner response: accepted. The task boundary and inventory publication gate
  separate local creation from public shipping.
- Disposition: mitigated
